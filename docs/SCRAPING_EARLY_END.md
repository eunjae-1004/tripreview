# 스크래핑이 두 개 기업만 하고 종료되는 경우

## 코드 상의 동작

- **기업 개수 제한은 없습니다.** `SELECT * FROM companies` 로 전체 기업을 가져와 `for (const company of companies.rows)` 로 모두 순회합니다.
- 2개만 처리하고 끝나는 **명시적 로직(예: slice(0, 2), break)** 은 없습니다.

---

## 가능한 원인

### 1. 실제로 기업이 2개뿐인 경우

- DB `companies` 테이블에 행이 2개만 있으면, 2개만 처리 후 **정상 완료**됩니다.
- **확인**: 작업 시작 로그에 `기업 N개, 포털: ...` 가 찍힙니다. N이 2이면 기업이 2개만 있는 것입니다.

### 2. 사용자 중지

- 화면에서 **중지** 버튼을 누르면 `cancelRequested` 가 true 가 되고, 다음 기업 시작 시 `ensureNotCancelled()` 에서 `JobCancelledError` 가 발생합니다.
- 이 에러는 **다음 기업으로 넘어가지 않고** 작업 전체를 종료시키며, 작업 상태는 **stopped** 로 저장됩니다.
- **확인**: 작업 상태가 `stopped` 이고, 진행 로그/에러 메시지에 "사용자에 의해 작업이 중지" 가 있으면 이 경우입니다.

### 3. 플랫폼 타임아웃 (Railway 등)

- 호스팅 환경에서 **요청/워커 실행 시간 제한**(예: 5분, 10분)이 있으면, 시간 초과 시 프로세스가 강제 종료됩니다.
- 2개 기업 × (네이버/카카오/야놀자/구글 등) 으로 시간이 길어지면, 2개 처리 후 구간에서 타임아웃이 날 수 있습니다.
- **확인**: 작업 상태가 **completed** 도 **failed** 도 아닌 채로 멈추거나, Railway 로그에 timeout/kill 메시지가 있는지 봅니다.

### 4. 메모리 부족(OOM)

- 리뷰가 많거나 브라우저가 오래 돌면 메모리 사용량이 커져, 플랫폼이 프로세스를 강제 종료할 수 있습니다.
- **확인**: 호스팅 로그에서 OOM(Out of Memory) 또는 process killed 메시지를 확인합니다.

### 5. 예외로 인한 작업 실패

- **기업 단위** try/catch 가 있어서, 한 기업에서 예외가 나면 로그를 남기고 **다음 기업으로 진행**합니다.
- 반면 **작업 전체**를 감싼 try/catch 밖에서 나는 에러(예: `ensureNotCancelled()` 의 `JobCancelledError` 재throw, 또는 초기화 실패)는 **전체 작업을 멈추고** 상태를 **failed** 또는 **stopped** 로 만듭니다.
- **확인**: `scraping_jobs` 테이블의 `error_message` 와 작업 상태(`status`)를 확인합니다.

## 확인 방법 요약

| 확인 항목 | 방법 |
|-----------|------|
| 기업이 실제로 몇 개인지 | 작업 시작 시 로그: `기업 N개, 포털: ...` (또는 진행 로그: `기업 1/N`, `기업 2/N` …) |
| 정상 완료인지 | 진행 로그에 `모든 기업 처리 완료 (총 N개)` 가 있고, 상태가 **completed** |
| 사용자 중지인지 | 상태 **stopped**, 에러 메시지에 "중지" |
| 타임아웃/크래시인지 | 상태가 completed/stopped/failed 로 바뀌지 않음, 또는 호스팅 로그에 timeout/OOM/kill |

---

## 데이터가 많아서인지 여부

- **코드만 보면** “데이터가 많아서 2개만 하도록 제한했다”는 부분은 **없습니다**.
- 다만 **데이터가 많으면** 한 기업당 처리 시간이 길어져서, 위의 **3(타임아웃)** 이나 **4(OOM)** 가 일어나 “2개만 하고 종료”처럼 보일 수는 있습니다.
- 따라서 **진행 로그**(기업 1/N, 2/N, …)와 **작업 상태·에러 메시지**, **호스팅 로그**를 함께 보면 원인 구분이 가능합니다.
